# Implementation Plan: SPEC-04 — Python App Consolidation

**Branch**: `20260510-145455-python-app-consolidation` | **Date**: 2026-05-10 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/Users/nhan.ngo/rmit/RMIT-Web-based-Data-Application/specs/20260510-145455-python-app-consolidation/spec.md`

## Summary

Retire the Go `backend/` service and merge `backend/` + `ai-service/` into a single unified Python `app/` service. Replace `scripts/reindex_products.go` and `scripts/search_relevance_check.sh` with Python equivalents in `app/scripts/`. Update `docker-compose.yml` to use an `app` service and update Makefile targets to drop the Go toolchain dependency.

**Amendment (post-initial implementation)**: Move `migrations/` into `app/migrations/` and replace the `golang-migrate` CLI dependency with a pure-Python migration runner (`app/scripts/migrate.py`). Also replace the `psql` CLI dependency for seeding with `app/scripts/snapshotdb.py`. This eliminates all non-Python host tool dependencies from the developer workflow.

**Constraints**: Must keep existing `.env` variable names; `make migratedb` must still work; existing SQL migration files must not be modified; `golang-migrate` and `psql` CLI tools are no longer required on the host

## Action Items (Phase-by-phase)

### Phase A — `app/` package scaffold

1. Create `app/` with all `__init__.py` stubs and `requirements.txt`.
2. Create `app/Dockerfile` using `python:3.11-slim` + `uv`.
3. Create `app/config.py` reading from `.env` (DATABASE_URL scheme conversion, PORT, OPENSEARCH_URL, etc.).
4. Create `app/database.py` with SQLAlchemy async engine and session dependency.
5. Create `app/exceptions.py` with sentinel exception classes.

### Phase B — SQLAlchemy ORM models

1. Create `app/models/product.py` (Product + Photo).
2. Create `app/models/user.py` (User).
3. Create `app/models/review.py` (Review).
4. Create `app/models/order.py` (Order + OrderItem).
5. Export all models from `app/models/__init__.py`.

### Phase C — FastAPI app + health endpoint

1. Create `app/routers/health.py` with `GET /api/v1/health`.
2. Create `app/main.py` with app factory, lifespan (DB + OS client init), router registration.
3. Smoke-test locally: `uvicorn app.main:app` → `curl /api/v1/health`.

### Phase D — opensearch-py helpers (port from Go)

1. Create `app/opensearch/client.py` with singleton and `get_opensearch` FastAPI dependency.
2. Port `backend/opensearch/query_builder.go` → `app/opensearch/query_builder.py`.
3. Port `backend/opensearch/keyword_search.go` → `app/opensearch/keyword_search.py`.
4. Port `backend/opensearch/semantic_search.go` → `app/opensearch/semantic_search.py`.
5. Port `backend/opensearch/fallback_logic.go` → `app/opensearch/fallback_logic.py`.

### Phase E — Python operational scripts (port from Go/Bash)

1. Create `app/scripts/reindex_products.py`:
   - Port `scripts/reindex_products.go` logic using `psycopg2` / `sqlalchemy` for DB and `opensearch-py` for bulk API.
   - Include `placeholder_unit_vector_384(seed_text)` using `hashlib.sha256`.
   - Output format: `indexed=N failed=M skipped=K`.
2. Create `app/scripts/search_relevance_check.py`:
   - Port `scripts/search_relevance_check.sh` using `opensearch-py`.
   - Print per-query hit count in same format as shell script.
3. Deprecate old scripts:
   - Add `# DEPRECATED: use app/scripts/reindex_products.py` to `scripts/reindex_products.go`.

### Phase F — Docker Compose update

1. Remove `backend` service (or comment out).
2. Uncomment / add `app` service: build from `./app`, ports `8080:8080`, env overrides for container DNS, healthcheck on `/api/v1/health`.
3. Remove `ai-service` service (already commented out; clean up).
4. Verify `docker compose up -d` starts exactly 3 services.

### Phase G — Makefile update

1. Replace `reindexproducts` target: `python app/scripts/reindex_products.py` (remove `go mod tidy && go run`).
2. Add optional `relevancecheck` target: `python app/scripts/search_relevance_check.py`.
3. Add `alias` `reindex-products: reindexproducts` for backward compat.
4. Add `dockerbuildapp` and `dockerrunapp` convenience targets if useful.

### Phase I — Python migration runner + snapshot loader (Amendment)

1. Move `migrations/` directory into `app/migrations/` (preserving all SQL files unchanged).
2. Move `scripts/snapshot_seed.sql` into `app/scripts/snapshot_seed.sql`.
3. Create `app/scripts/migrate.py`:
   - Tracks applied migrations in a `schema_migrations` table (auto-created).
   - Reads `NNNNNN_name.up.sql` / `NNNNNN_name.down.sql` files from `app/migrations/`.
   - Supports subcommands: `up` (apply all pending), `down <N>` (roll back N steps), `reapply` (drop all + up).
   - Exits non-zero with clear error messages.
4. Create `app/scripts/snapshotdb.py`:
   - Reads `app/scripts/snapshot_seed.sql` and executes it via psycopg2.
   - Supports the same `.env` / `DATABASE_URL` loading as other scripts.
5. Update `Makefile`: replace `migrate` CLI calls with `python app/scripts/migrate.py` and `psql` call with `python app/scripts/snapshotdb.py`.
6. Update `migrations/` path references in `docker-compose.yml` comments and docs.

### Phase H — `.env` and documentation updates

1. Update `.env.example` `DATABASE_URL` comment to note both `postgres://` (golang-migrate) and `postgresql+asyncpg://` (SQLAlchemy) usage.
2. Update `README.md` "where to change what" table to reference `app/` instead of `backend/` + `ai-service/`.
3. Update `docs/repository-layout.md` tree.
4. Update `docs/layering.md` and `docs/layering-audit.md` for Python layering.
